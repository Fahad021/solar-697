Irradiance and Meteorological

Goal

The primary goal of the solar data collection process is to provide quantitative irradiance data for the purpose of estimating photovoltaic electric power generation in a geospatial-context. Secondarily, and of nearly equal importance, is the extraction of trends from the raw data, which can be used to enhance understanding of future trends in solar irradiance and electric power generation potential. Lastly, to incorporate forecasting using one or more models with the goal of providing actionable estimates of future irradiance quantities and to a lesser extent, envisioning future meteorological conditions. Using the irradiance forecast data, estimates of average power generation potential would be available for residential customer photovoltaic investment decision-making. Each of these data product features will be wrapped and presented via the Solar Irradiance Data Explorer dashboard (SIDE-dash).

Background

The central concern with photovoltaic solar power generation is the total available solar irradiance for a given location or region. The absolute availability depends on the calendar season and the latitude of the location. Summertime of the northern hemispheres places the sun higher in the sky and thereby reduces zenith angle or angle of incidence of solar radiation arriving at earth’s upper atmosphere and ultimately the surface of the earth. The decreased zenith angle also reduces the path length through the atmosphere and the corresponding attenuation of available irradiance. As the sun is lower in the wintertime sky, the result is an increase in zenith angle and greater attenuation of available irradiance. An increase in latitude also increases the zenith angle with effects like that of seasonal wintertime.

Meteorological conditions including atmospheric water (Tasie, Banyie, & israel-cookey, 2018) and fine particulate matter also plays a significant role in terrestrial-based photovoltaic electric power production. As light passes through the earth’s atmosphere, varying levels of water vapor (precipitable water) or cloud cover will cause varying levels of diffusion of sun light (wikipedia.com, 2021). Diffusion has the effect of increased zenith as the radiation is no longer straight-line from the sun source, but rather arriving at a higher angle of incidence due to diffusion’s scattering effects. Particulate matter can have a similar effect if not an outright blockage of the sun’s rays. All these factors combine to create a highly seasonal response to solar power production.

Data

The primary source of solar irradiance data is obtained from the National Renewable Energy Laboratory (NREL, 2021) and the National Solar Radiation Database (NSRDB). NREL is an organization of the U.S. Department of Energy and a globally respected source of renewable energy data, recommended by industry and academia sources alike. More specifically, the on-demand data was requested through the NREL provided API which allows client-specific location, timeframe, and data feature requests. The NSRDB, provided by NREL, is a 30-minute time increment, 4-km spatially gridded dataset with complete coverage of North America, and more. For each of the ZIP Code-based locations requested, all data fields and all twenty-three years of 1-hour increment data (1998-2020) were downloaded. The full data set include the following fields:

·       date_time, zipcode, location_id, Year, Month, Day, Hour

·       Clearsky_DHI, DHI, Clearsky_DNI, DNI, Clearsky_GHI, GHI

·       Temperature, Dew_Point, Relative_Humidity, Precipitable_Water

·       Pressure, Wind_Direction, Wind_Speed, Cloud_Type, Fill_Flag

·       Solar_Zenith_Angle, Surface_Albedo

·       Global_Horizontal_UV_Irradiance_(280-400nm)

·       Global_Horizontal_UV_Irradiance_(295-385nm)]

Each of these fields is retained in the project’s Zip Code request databases, but not in the current monthly aggregated database. This is mostly due to file size restrictions imposed by GitHub. While not a part of the project GitHub repository, all requested raw data downloads have been retained for future use as necessary.

The current ZIP Code spatial data was extracted from the Gazetteer Files page of U.S. Census Bureau (census.gov, 2021). The Gazetteer data includes a geo-spatially centered latitude/longitude coordinate for each ZIP Code polygon. The ZIP Code geo-coordinate was used as the location parameter in the NREL API queries of the NSRDB dataset. Additional fields in the Gazetteer data include land and water area for each ZIP Code. This data could have future use in determining if additional NSRDB coverage is necessary given the spatial extents of a given ZIP Code.

A third and final data source is UnitedStatesZipCodes.org which has available an up-to-date dataset (UnitedStatesZipCodes.org, 2021) that relates ZIP Codes with geo-political features such as city, county, state, and more. This dataset along with the Gazetteer data formed the basis for the ‘geo_zipcodes’ table used in the project database(s).

Approach

As a matter of portability and resource minimization, SQLite3 was determined to be the database of choice for the project. SQLite3 is a part of the Python Standard Library and functional on all compute platforms which support Python.

The ‘geo_zipcodes’ database was an early creation to be used as feedstock in the production of the actual irradiance database. Certain fields within the ‘geo_zipcodes’ database are updated with each download of new irradiance data. When a new irradiance database is built, the current ‘geo_zipcodes’ database is incorporated as the ‘geo_zipocodes’ table.

When a Zip Codes-based request for irradiance data is made from a team member, the ‘get_zipcode_geos.ipynb’ notebook script compiles the ZIP Codes into a simple YAML text file that is ingested by the ‘nsrdb_download.ipynb’ notebook script. As each NSRDB request arrives, minor data manipulations are performed, and the data is then written to the new ZIP Code irradiance database. Simultaneously the internal ‘geo_zipcodes’ table is updated as is the external database for future use.

Because all facets of the project operate at the monthly level, the choice to aggregate working data at the monthly level was an easy choice. Originally this operation was performed in real-time once the data was ingested from the database. As SIDE-dash took shape, irradiance data ingestion performance failed to meet expectations as twenty-three years of hourly data resulted in nearly 201,500 records.

As a late addition to the project, the ‘nsrdb_aggregator.ipynb’ notebook script was authored to reduce the overall size of the dataset while still retaining current functionality. The reduction in data volume was profound; a 43 ZIP Code irradiance database of raw data consumed 1.1GB of filesystem space. The aggregated hourly average by month database, of the same ZIP Codes, was reduced to 5.9MB.

An example of hourly average by month; ZIP Code 10514 (Chappaqua, New York), for the month of January 1998, has as Global Horizontal Irradiance (GHI) value of 75.3 Wh/m^2 (Watt-hours per square meter); meaning that on average, for every hour of the 24-hour day, 75.3 Wh/m^2 of solar radiation arrived at the earth’s surface in Chappaqua. As a point of reference Phoenix, Arizona received, on average, 144.8 Wh/m^2 per hour for the same month. Obviously, the sun does not shine 24-hours a day in these regions therefore the total irradiance (1,807.8 and 3475.2 respectively) would be captured during normal daylight hours for the location in question.

The dashboard currently includes two forecasting models built from the PMDARIMA open-source time-series package. The first model performs a Fast Fourier Transform (FFT) of the data to eliminate the seasonal component of the data; a necessary and important step for any time-series forecast model. A simple model pipeline is implemented to ensure that the transform is correctly applied to all data, be it training, validation or test. The second model relies on more traditional seasonal decomposition (parameters P, D, Q) to eliminate seasonality from the data. Both models incorporate standard Order parameters p, d, and q for auto-regressive (AR), differencing (Integration), and moving average (MA). The benefit of the PMDARIMA package is the automatic Order optimization methods that are built into the library. In the context of SIDE-dash, each forecast is optimized for the specific feature selected. An interesting note is the FFT model is typically three to five times faster than the traditional seasonal decomposition model. The metric Root Mean Squared Error (RMSE) was chosen as the model test performance indicator and comparator. The advantage of RMSE is that the calculated value retains the original units of the training and forecast data, simplifying its interpretation and comparison.

Outcome

The primary goal of the irradiance data collection process is the production of a database(s) product that all team members can easily query and retrieve irradiance data for application to their respective needs. The outcome of this goal is realized through the Solar Irradiance Data Explorer dashboard that provides multiple exploratory views of the aggregated data, decomposed trend data, and time-series forecasting of the data. While the irradiance data and forecast would see the most direct use, the trend data offers a tantalizing perspective of future potential.
